Prototipos funcionales con IA desde cero

Introducción

En un contexto donde la inteligencia artificial permite acelerar significativamente el desarrollo de soluciones, surge la necesidad de contar con una guía clara que permita transformar una idea en un prototipo funcional en el menor tiempo posible.

Este documento tiene como objetivo brindar un paso a paso simple y práctico para crear prototipos utilizando herramientas de IA, sin necesidad de una estructura técnica compleja ni conocimientos avanzados de desarrollo.

Está orientado principalmente a perfiles como Product Owners, Analistas y Equipos de negocio que necesitan validar ideas, construir pruebas de concepto o generar primeras versiones funcionales para compartir con stakeholders.

A lo largo de esta guía se recorrerán los pasos para:

  • Preparar el entorno de trabajo
  • Configurar herramientas clave
  • Utilizar inteligencia artificial dentro del flujo de desarrollo
  • Generar un prototipo full-stack funcional (frontend + backend)
  • Publicar y compartir una URL pública
💡 La premisa principal es: pasar de una idea a algo funcional lo más rápido posible, utilizando la IA como acelerador del proceso.

1. Crear cuenta de GitHub

Antes de comenzar, es imprescindible contar con una cuenta de GitHub. GitHub no solo se utiliza para trabajar con repositorios, sino que también es el punto de acceso a herramientas de inteligencia artificial como GitHub Copilot. Sin una cuenta activa y correctamente configurada, no será posible utilizar IA dentro del entorno de desarrollo.

Pasos:

  1. Ingresar a https://github.com
  2. Crear una cuenta con email corporativo
  3. Verificar el email
  4. Completar el perfil básico
💡 Se debe utilizar una cuenta asociada a la organización si se requiere acceso a licencias de Copilot.

1.1 Unirse a la organización de la empresa en GitHub

Para poder utilizar correctamente las herramientas de la organización (incluyendo IA), es necesario formar parte de la organización en GitHub.

¿Qué necesito?

  • Tu cuenta de GitHub creada
  • Una invitación del administrador (Tech Lead o DevOps)

Instrucciones:

  1. Enviarle a tu Tech Lead o DevOps tu username de GitHub
  2. El administrador enviará una invitación por email
  3. Abrir el email y hacer click en "Accept invitation"
  4. Confirmar el acceso a la organización
  5. Hacer click en "Join [nombre de la organización]"

7. Definir estructura del proyecto

7.1 ¿Un repositorio o dos?

Es una duda común al iniciar un prototipo. La decisión depende del tamaño del equipo y los objetivos del proyecto.

7.2 Recomendación: monorepo

Estructura sugerida:

/src
  /backend
  /frontend

✔️ Ventajas:

  • Más simple de gestionar
  • Más rápido de configurar
  • Ideal para prototipos y validaciones

⚠️ Cuándo separar en repos distintos:

  • Equipos independientes que trabajan en paralelo
  • Deploys en infraestructuras separadas
  • Proyecto en crecimiento con necesidades distintas
💡 Recomendación: mantener monorepo durante toda la etapa de prototipado.

9. Definir features con el agente analista

Una vez que el prototipo base está generado, cada nueva funcionalidad sigue un proceso estructurado antes de escribir una sola línea de código. El responsable de ese proceso es el agente analista.

9.1 ¿Qué es una Historia de Usuario (HU)?

Una Historia de Usuario es un documento que describe con precisión qué debe hacer una feature, desde el punto de vista del usuario, antes de que alguien empiece a implementarla. Sirve como contrato entre quien pide la feature y quien la construye.

Una buena HU responde estas preguntas:

  • ¿Quién lo necesita y para qué?
  • ¿Qué pasos sigue el usuario?
  • ¿Qué datos entran y qué resultado se espera?
  • ¿Qué pasa cuando algo sale mal?
  • ¿Qué queda explícitamente fuera del alcance?
💡 El objetivo es que cualquier desarrollador (o agente de IA) pueda implementar la feature sin necesidad de hacer preguntas adicionales.

9.2 Cómo invocar al analista

En el chat de VS Code, escribir @analista seguido de la idea en lenguaje natural:

@analista quiero agregar un formulario de contacto
@analista necesito que los usuarios puedan exportar sus resultados en PDF
@analista me gustaría tener un historial de consultas recientes

No hace falta dar detalles técnicos. El agente hace las preguntas necesarias en rondas progresivas:

  1. Ronda 1 — Entender el valor: qué necesita el usuario y por qué
  2. Ronda 2 — Entender el flujo: paso a paso, datos de entrada y resultado esperado
  3. Ronda 3 — Casos borde: errores, validaciones y restricciones
  4. Ronda 4 — UI/UX: dónde aparece, estados de carga y error
⚠️ El agente nunca genera código. Su único output es la HU. Tampoco avanza sin aprobación explícita del usuario.

9.3 El flujo completo: idea → HU → código

PasoQuién actúaQué ocurre
1. IdeaUsuarioDescribe la feature en lenguaje natural a @analista
2. Entrevista@analistaHace preguntas en rondas para entender el requerimiento
3. HU generada@analistaPresenta la HU completa y pide aprobación
4. RevisiónUsuarioAprueba o pide ajustes. Se itera hasta que quede ok
5. AprobaciónUsuarioConfirma con "aprobado", "dale" o "sí"
6. HU guardada@analistaGuarda el archivo en .ai/stories/ e indica cómo implementar
7. Implementación@backend / @web-builderLeen la HU aprobada y generan el código

9.4 Ejemplo de HU generada

Una HU generada por el analista tiene esta estructura:

HU-007 — Exportar carta astral como imagen

Historia de usuario
Como usuario que generó su carta astral,
quiero poder descargarla como imagen PNG,
para compartirla fácilmente en redes sociales.

Flujo principal
1. El usuario visualiza su carta astral generada
2. Hace click en el botón "Descargar"
3. El sistema genera una imagen con los datos de la carta
4. El navegador descarga automáticamente el archivo PNG

Criterios de aceptación
Backend
- [ ] Endpoint POST /api/cartas/:id/imagen que devuelve la imagen en base64
Frontend
- [ ] Botón "Descargar" visible solo cuando hay carta generada
- [ ] Estado de carga mientras se genera la imagen
- [ ] Si falla, mostrar mensaje de error claro

Casos borde
- Sin carta generada → botón oculto o deshabilitado
- Error en servidor → mensaje "No se pudo generar la imagen"

Fuera de alcance
- No incluye compartir directo a redes (solo descarga local)
💡 Esta precisión es lo que permite que el agente de IA implemente la feature correctamente de primera, sin ir y volver con preguntas.

10. Recomendaciones para trabajar con IA

  • ✔️ Pedir soluciones simples — evitar pedir todo de una vez
  • ✔️ Iterar rápido — construir de a partes pequeñas y verificar
  • ✔️ Validar cada parte — no asumir que el código funciona sin probarlo
  • ✔️ Evitar complejidad innecesaria — la IA tiende a sugerir más de lo necesario
  • ✔️ Dar contexto claro — cuanto mejor sea el prompt, mejor el resultado
💡 La IA acelera, pero el criterio sigue siendo humano.

11. Configurar base de datos en la nube (Turso)

Por defecto, el prototipo usa SQLite local, que funciona en la computadora pero se pierde con cada nuevo deploy en Railway o Render. Para tener persistencia real en producción, se usa Turso: una base de datos SQL en la nube compatible con SQLite, gratuita para prototipos.

Paso 1 — Crear cuenta y base de datos (vos)

  1. Ir a https://turso.tech → Sign up (se puede usar la cuenta de GitHub)
  2. Click en Create Database → elegir un nombre y una región → click en Create
  3. Dentro de la base creada, ir a la pestaña Connect
  4. Copiar la Database URL (formato: libsql://nombre-usuario.turso.io)
  5. Click en Generate Token → copiar el token
⚠️ El token se muestra una sola vez. Guardarlo antes de cerrar la pantalla. No compartirlo por el chat.

Paso 2 — Configurar el proyecto (agente)

Una vez que tenés la URL y el token, decile al agente:

configurame el .env de Turso. La URL es [pegar URL] y el token es [pegar token]

El agente crea el archivo .env con las variables correctas. Como el .env ya está en el .gitignore, no se sube a GitHub.

Paso 3 — Agregar las variables en Railway (vos)

  1. Ir a Railway → proyecto → servicio → tab Variables
  2. Click en New Variable y agregar:
    • TURSO_URL → la URL copiada de Turso (formato libsql://...)
    • TURSO_AUTH_TOKEN → el token generado
  3. Click en Deploy — Railway redeploya con las nuevas variables
💡 Con estas variables configuradas, la aplicación en producción guarda y lee datos de Turso igual que SQLite local, pero de forma persistente entre deploys.

12. Subir al repositorio y publicar

Paso 1 — Crear repositorio en GitHub

  1. Ir a https://github.com/new
  2. Ingresar un nombre para el repositorio
  3. Dejarlo vacío (sin README)
  4. Click en "Create repository"

Paso 2 — Subir el código

El agente de IA puede ejecutar estos comandos directamente. Solo hace falta indicarle la URL del repositorio recién creado:

git init
git add .
git commit -m "initial commit"
git remote add origin https://github.com/[usuario]/[repositorio].git
git branch -M main
git push -u origin main

Paso 3 — Publicar con Railway

Railway es una plataforma de hosting en la nube que despliega aplicaciones Node.js con configuración mínima y genera una URL pública.

  1. Ir a https://railway.app → Login with GitHub
  2. Click en New Project → Deploy from GitHub repo
  3. Si el repositorio no aparece, click en el engranaje Configure GitHub App:
    • En "Repository access" elegir "Only select repositories"
    • Buscar y seleccionar el repositorio → Click en "Save"
    • Volver a Railway y click en "Refresh"
  4. Seleccionar el repositorio
  5. En la pestaña Variables, agregar: NODE_ENV = production
  6. Esperar el primer deploy (~3-4 minutos). Ver progreso en la pestaña Deployments
  7. Ir a Settings → Networking → Generate Domain para obtener la URL pública

Alternativa: Render

  1. Ir a https://render.com → New → Web Service
  2. Conectar GitHub y seleccionar el repositorio
  3. Configurar: Build Command: npm run build / Start Command: npm start
  4. Click en Create Web Service → esperar deploy → copiar URL
⚠️ El plan gratuito de Render "duerme" el servidor tras 15 minutos sin tráfico (la primera solicitud tarda ~30 segundos). Railway no tiene ese comportamiento pero tiene un límite de crédito mensual.

13. Flujo de actualizaciones

Cada vez que se quiera modificar algo del prototipo y actualizar la URL pública, simplemente se le indica al agente de IA el cambio deseado en lenguaje natural:

💡 Ejemplos: "cambiá el color del botón", "agregá un campo de teléfono", "el título dice X, cambialo por Y"

El agente realiza los cambios en el código y sube todo al repositorio automáticamente. Railway o Render detectan el push y actualizan la URL pública en aproximadamente 3 minutos, sin intervención manual.

Cierre

El objetivo de esta guía es permitir pasar de una idea a un prototipo funcional en el menor tiempo posible, con el menor esfuerzo de configuración.

Este enfoque permite:

  • Validar ideas rápidamente con stakeholders reales
  • Mostrar avances concretos antes de comprometer recursos
  • Iterar con bajo costo técnico y organizacional
